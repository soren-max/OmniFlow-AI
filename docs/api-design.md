# API Design

## Current Stage

Currently implemented endpoints:

```
GET   /                            → Service info
GET   /health                      → Service health status
POST  /api/projects                → Create a content project
GET   /api/projects/{id}           → Get a content project by id
POST  /api/projects/{id}/preview   → Generate platform previews
POST  /api/projects/{id}/agent-preview → Run experimental LangGraph preview skeleton
POST  /api/projects/{id}/publish   → Mock publish selected platforms
GET   /api/runs/{run_id}           → Get an Agent Run trace record
GET   /api/runs/{run_id}/steps     → List Agent Step trace records for a run
GET   /docs                        → Swagger UI (auto-generated by FastAPI)
```

Projects are stored in memory (in-memory repository). Previews are generated
using the PlatformAdapter mock adapters. Mock publish is supported through
`mock_publish`; no real platform API is called. LangGraph is present only as a
deterministic preview workflow skeleton and does not call a real LLM.

Agent Run and Agent Step endpoints are read-only at this stage. They expose the
in-memory trace data model used by `AgentTraceService`; they do not start LangGraph
workflows or execute Agent nodes.

Supported preview platforms:

- `wechat` — WeChat Official Accounts / 微信公众号
- `zhihu` — Zhihu / 知乎
- `bilibili` — Bilibili / B站
- `xiaohongshu` — Xiaohongshu / 小红书
- `douyin` — Douyin / 抖音

## API Design Principles (Future)

The API will follow RESTful conventions:

- Use nouns for resources: `/projects`, `/contents`, `/runs`.
- Use HTTP methods for actions: GET, POST, PUT, DELETE.
- Use consistent pagination: `?page=1&size=20`.
- Use a consistent response envelope for core APIs.
- Use stable machine-readable error codes.
- Use Pydantic for request/response validation.
- Version via URL prefix if breaking changes are needed.

## Current Endpoints

### Projects

| Method | Path | Description |
|--------|------|-------------|
| POST | /api/projects | Create a new content project |
| GET | /api/projects/{id} | Get project details and saved previews |
| POST | /api/projects/{id}/preview | Generate previews for target platforms |
| POST | /api/projects/{id}/agent-preview | Run experimental deterministic LangGraph preview |
| POST | /api/projects/{id}/publish | Mock publish selected target platforms |

### Agent Runs

| Method | Path | Description |
|--------|------|-------------|
| GET | /api/runs/{run_id} | Get an Agent Run trace record |
| GET | /api/runs/{run_id}/steps | List Agent Step trace records for a run |

Trace records are currently created by the service layer and stored in memory.
These endpoints are primarily for inspecting traces once future workflow code writes
Agent Run and Agent Step records.

### Create Project — `POST /api/projects`

```json
// Request
{
  "title": "My Article",
  "source_text": "Full article body...",
  "source_url": "https://example.com/original"  // optional
}

// Response (201 Created)
{
  "success": true,
  "data": {
    "id": "a1b2c3d4e5f6",
    "title": "My Article",
    "source_text": "Full article body...",
    "source_url": "https://example.com/original",
    "status": "created",
    "created_at": "2025-01-01T00:00:00Z",
    "previews": []
  },
  "error": null
}
```

### Mock Publish — `POST /api/projects/{id}/publish`

```json
// Request
{
  "target_platforms": ["wechat", "zhihu", "bilibili", "xiaohongshu", "douyin"],
  "mode": "mock"
}

// Response (200 OK)
{
  "success": true,
  "data": {
    "project_id": "a1b2c3d4e5f6",
    "mode": "mock",
    "results": [
      {
        "platform": "douyin",
        "platform_display_name": "Douyin",
        "status": "success",
        "mock_url": "mock://douyin/post/mock-123",
        "message": "Mock published to Douyin successfully.",
        "metadata": {
          "mock": true,
          "platform": "douyin"
        }
      }
    ],
    "published_at": "2025-01-01T00:00:00Z"
  },
  "error": null
}
```

Only `mode: "mock"` is supported. `mode: "real"` returns
`REAL_PUBLISH_NOT_SUPPORTED` because real publishing requires future Human Review,
explicit user authorization, and secure platform credentials.

`GET /api/runs/{run_id}` returns an `AgentRun` record. `GET /api/runs/{run_id}/steps`
returns the ordered `AgentStep` records for that run. Both use the standard response
envelope. Preview and Mock Publish do not write trace records yet; a later LangGraph
workflow PR should map those operations to Agent Step records.

### Agent Preview — `POST /api/projects/{id}/agent-preview`

This experimental endpoint runs the deterministic LangGraph preview skeleton.
It returns workflow state with normalized input, platform strategy, previews,
errors, and status. It does not replace `/preview`, call real LLMs, or perform
real publishing. Platform previews are still generated through the PlatformAdapter
registry.

## Error Format

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "PROJECT_NOT_FOUND",
    "message": "Project not found: a1b2c3d4e5f6",
    "details": {
      "project_id": "a1b2c3d4e5f6"
    }
  }
}
```

Current error codes include:

- `PROJECT_NOT_FOUND`
- `INVALID_PLATFORM`
- `ADAPTER_NOT_FOUND`
- `ADAPTER_EXECUTION_FAILED`
- `REAL_PUBLISH_NOT_SUPPORTED`
- `UNSUPPORTED_PUBLISH_MODE`
- `VALIDATION_ERROR`
- `HTTP_ERROR`

## Currently Not Implemented

- **Database persistence** — projects are stored in memory only.
- **Real platform publishing** — only mock preview generation and mock publish results are implemented.
- **Project update/delete** — only create, read, preview, and mock publish exist.
- **Listing all projects** — no paginated list endpoint.
- **Authentication/authorization** — no user or API key system.
- **Agent workflow** — content adaptation is done directly via adapter, not via LangGraph.
