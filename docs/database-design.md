# Database Design

## Current Stage

The backend now uses SQLAlchemy repositories with Alembic-managed PostgreSQL
migrations for core recovery and audit data. Database configuration comes from
`DATABASE_URL`; local development uses the PostgreSQL service in `docker-compose.yml`.

Persisted in this PR:

- `ContentProject`
- `PlatformContent` / preview result rows
- `PublishTask` / mock publish result rows
- `EvaluationReport`
- `AgentRun`
- `AgentStep`
- Project-level Human Review status via `content_projects.status`

Planned for later PRs:

- `ReviewRecord`
- `EvaluationResult`
- User feedback and production observability records

The application still does not call real LLMs and does not perform real platform
publishing.

## Current Tables

### content_projects

Stores the source content project submitted through `POST /api/projects`.

```text
id
title
source_text
source_url
status
created_at
updated_at
```

`status` is currently used as the Human Review gate for Mock Publish:

- `created` after project creation
- `pending` after preview generation
- `approved` after reviewer approval
- `rejected` after reviewer rejection

This is intentionally project-level state. A dedicated `ReviewRecord` table can
be added later when the review workflow needs reviewer identity, comments, or an
audit history.

### platform_contents

Stores platform-specific preview/adaptation output generated by the mock
PlatformAdapter preview pipeline.

```text
id
project_id
platform
platform_display_name
title
content
metadata_json
preview_json
validation_json
rendered_html
word_count
estimated_read_time_min
warnings_json
created_at
```

### publish_tasks

Stores one mock publish operation per project request.

```text
id
project_id
mode
status
created_at
```

### publish_results

Stores per-platform mock publish results for a publish task.

```text
id
task_id
platform
platform_display_name
status
mock_url
message
metadata_json
```

### evaluation_reports

Stores the latest and historical rule-based quality evaluation reports for
project previews. The current evaluator is deterministic and does not call an
LLM.

```text
id
project_id
average_score
platform_scores_json
issues_json
suggestions_json
created_at
```

### agent_runs

Stores one deterministic LangGraph preview workflow execution.

```text
run_id
project_id
workflow_name
status
input_snapshot
output_snapshot
error_message
started_at
finished_at
total_latency_ms
created_at
```

### agent_steps

Stores one wrapped LangGraph node execution within a run.

```text
step_id
run_id
node_name
status
input_snapshot
output_snapshot
tool_calls
error_message
latency_ms
started_at
finished_at
created_at
```

## Migrations

Alembic configuration lives at:

- `alembic.ini`
- `alembic/env.py`
- `alembic/versions/20260618_0001_create_core_persistence_tables.py`
- `alembic/versions/20260618_0002_create_evaluation_reports.py`

Apply migrations locally:

```bash
make docker-up
make db-upgrade
```

Without `make`:

```bash
docker compose up -d
uv run alembic upgrade head
```

Create a future migration after model changes:

```bash
make db-migrate message="describe change"
```

## Notes

JSON columns are used for flexible snapshots and adapter payloads:

- preview metadata
- preview payload
- validation payload
- run input/output snapshots
- step input/output snapshots
- tool call placeholders

Human Review persistence and richer evaluation artifacts should be added in
focused follow-up PRs once those modules need reviewer identity, comments,
datasets, or regression report history.
